# READYOU.md ‚Äî Carnatic guru-shishya knowledge graph: data schema and access methods

> **This file is addressed to agents operating in this repository.**
> All reads and writes to this dataset are performed through the CLI methods
> documented below. Agents must never edit JSON files directly.

---

## Governing principle

**Significance over completeness.** A musician belongs here if they materially
shaped the sound, transmission, or scholarship of the Carnatic tradition. Fringe
or obscure figures are excluded unless they are a necessary topological link
between two significant nodes.

---

## Data files

| file / directory | role |
|---|---|
| `musicians/{id}.json` | One bare JSON node object per musician (preferred) |
| `musicians/_edges.json` | All guru‚Üíshishya edges in a single array |
| `musicians.json` | **Legacy monolithic file** ‚Äî superseded; kept as `.bak` after migration |
| `ragas/{id}.json` | One bare JSON raga object per raga (~160 files) |
| `compositions/{id}.json` | One bare JSON composition object per composition (~130 files) |
| `compositions/_composers.json` | All composers in a single bare array (~50 entries) |
| `compositions.json` | **Legacy monolithic file** ‚Äî superseded; kept as `.bak` after migration |
| `recordings/*.json` | One bare JSON object per structured concert recording |
| `graph.json` | **Derived artefact** ‚Äî regenerated by `render.py`; never edit directly |

### musicians/ directory layout

```
carnatic/data/musicians/
‚îú‚îÄ‚îÄ _edges.json              ‚Üê all guru‚Üíshishya edges (array of edge objects)
‚îú‚îÄ‚îÄ tyagaraja.json           ‚Üê one file per musician node
‚îú‚îÄ‚îÄ ms_subbulakshmi.json
‚îú‚îÄ‚îÄ semmangudi_srinivasa_iyer.json
‚îî‚îÄ‚îÄ ...                      ‚Üê 68 files total (one per musician)
```

- Files prefixed with `_` are **not** treated as nodes (only `_edges.json` currently).
- Each `{id}.json` is a bare JSON object ‚Äî no wrapping array or key.
- The `id` field inside the file must match the filename (without `.json`).
- `_edges.json` is a bare JSON array of edge objects.

### ragas/ directory layout

```
carnatic/data/ragas/
‚îú‚îÄ‚îÄ abheri.json              ‚Üê one file per raga (bare JSON object)
‚îú‚îÄ‚îÄ bhairavi.json
‚îú‚îÄ‚îÄ kalyani.json
‚îú‚îÄ‚îÄ shankarabharanam.json
‚îî‚îÄ‚îÄ ...                      ‚Üê ~160 files total (72 melakartas + ~88 janya ragas)
```

- Each `{id}.json` is a bare JSON raga object ‚Äî no wrapping array or key.
- The `id` field inside the file must match the filename (without `.json`).
- Ragas are a **top-level concept**: they exist independently as RTPs, alapanas, etc.
  Not all ragas have compositions; not all ragas are melakartas.

### compositions/ directory layout

```
carnatic/data/compositions/
‚îú‚îÄ‚îÄ _composers.json          ‚Üê all composers as a bare JSON array (~50 entries)
‚îú‚îÄ‚îÄ entharo_mahanubhavulu.json  ‚Üê one file per composition (bare JSON object)
‚îú‚îÄ‚îÄ evari_mata_vintura.json
‚îî‚îÄ‚îÄ ...                      ‚Üê ~130 files total (one per composition)
```

- Files prefixed with `_` are **not** treated as compositions (`_composers.json` only).
- Each `{id}.json` is a bare JSON composition object ‚Äî no wrapping array or key.
- The `id` field inside the file must match the filename (without `.json`).
- `_composers.json` is a bare JSON array of composer objects (mirrors `_edges.json` pattern).

## Read methods ‚Äî `python3 carnatic/cli.py`

All reads go through `cli.py`. Never read JSON files directly for orientation.

```bash
# Orientation
python3 carnatic/cli.py stats

# Existence checks (exit 0 = found, exit 1 = not found)
python3 carnatic/cli.py musician-exists    <id_or_label>
python3 carnatic/cli.py raga-exists        <id_or_name>
python3 carnatic/cli.py composition-exists <id_or_title>
python3 carnatic/cli.py recording-exists   <id_or_title>
python3 carnatic/cli.py url-exists         <youtube_url>

# Lookup (compact summary; add --json for full object)
python3 carnatic/cli.py get-musician       <id>
python3 carnatic/cli.py get-raga           <id>
python3 carnatic/cli.py get-composition    <id>
python3 carnatic/cli.py gurus-of           <musician_id>
python3 carnatic/cli.py shishyas-of        <musician_id>
python3 carnatic/cli.py lineage            <musician_id>
python3 carnatic/cli.py recordings-for     <musician_id>
python3 carnatic/cli.py compositions-in-raga <raga_id>
python3 carnatic/cli.py concerts-for       <musician_id>
python3 carnatic/cli.py co-performers-of   <musician_id>
python3 carnatic/cli.py concerts-with      <musician_id_a> <musician_id_b>
python3 carnatic/cli.py concert            <recording_id>  [--json]

# Melakarta / Cakra traversal (ADR-021)
# Exit 0 = is a melakarta; exit 1 = janya or not found
python3 carnatic/cli.py is-mela            <raga_id>
# List all janya ragas whose parent_raga == mela_raga_id
python3 carnatic/cli.py janyas-of          <mela_raga_id>
# Print the parent mela raga for a janya raga
python3 carnatic/cli.py mela-of            <janya_raga_id>
# Print the cakra number and name for any raga (climbs to parent if janya)
python3 carnatic/cli.py cakra-of           <raga_id>
# List all 6 mela ragas in a given cakra (1‚Äì12)
python3 carnatic/cli.py melas-in-cakra     <cakra_number>

# Validation (run after any write session)
python3 carnatic/cli.py validate
```

## Write methods ‚Äî `python3 carnatic/write_cli.py`

All writes go through `write_cli.py`. Every command:
- Validates referential integrity before writing.
- Writes atomically (temp file + rename).
- Prints a change-log line to stdout.
- Exits 0 on success or duplicate-skip; exits 1 on validation failure.

`write_cli.py` writes to `musicians/{id}.json` + `musicians/_edges.json` (directory mode)
or `musicians.json` (legacy fallback); and to `ragas/{id}.json`, `compositions/{id}.json`,
`compositions/_composers.json` (directory mode) or `compositions.json` (legacy fallback).
`graph.json` is a derived artefact ‚Äî never touched here.
**After any write session, run `python3 carnatic/render.py`.**

```bash
# ‚îÄ‚îÄ Musician graph writes ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ

python3 carnatic/write_cli.py add-musician \
    --id <id> --label <label> --era <era> \
    --instrument <inst> --source-url <url> \
    --source-label <label> --source-type <type> \
    [--born <year>] [--died <year>] [--bani <bani>]

python3 carnatic/write_cli.py add-edge \
    --source <guru_id> --target <shishya_id> \
    --confidence <float> --source-url <url> \
    [--note <text>]

python3 carnatic/write_cli.py add-youtube \
    --musician-id <id> --url <yt_url> --label <label> \
    [--composition-id <id>] [--raga-id <id>] \
    [--year <int>] [--version <text>]

python3 carnatic/write_cli.py add-source \
    --musician-id <id> --url <url> --label <label> --type <type>

python3 carnatic/write_cli.py remove-edge \
    --source <guru_id> --target <shishya_id>

python3 carnatic/write_cli.py patch-musician \
    --id <id> --field <field> --value <value>
# Permitted fields: label, born, died, era, instrument, bani  (id is immutable)

python3 carnatic/write_cli.py patch-edge \
    --source <guru_id> --target <shishya_id> \
    --field <field> --value <value>
# Permitted fields: confidence, source_url, note

# ‚îÄ‚îÄ Composition data writes ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ

python3 carnatic/write_cli.py add-raga \
    --id <id> --name <name> \
    --source-url <url> --source-label <label> --source-type <type> \
    [--aliases <csv>] [--melakarta <int>] \
    [--parent-raga <id>] [--notes <text>]

python3 carnatic/write_cli.py patch-raga \
    --id <raga_id> --field <field> --value <value>
# Permitted fields: name, parent_raga, melakarta, is_melakarta, cakra, notes
# (id and sources are immutable via this command)
# is_melakarta: use 'true' or 'false'
# cakra: integer 1‚Äì12 or 'null'
# melakarta: integer 1‚Äì72 or 'null'
# parent_raga: existing raga id or 'null'

python3 carnatic/write_cli.py add-composer \
    --id <id> --name <name> \
    --source-url <url> --source-label <label> --source-type <type> \
    [--musician-node-id <id>] [--born <year>] [--died <year>]

python3 carnatic/write_cli.py add-composition \
    --id <id> --title <title> \
    --composer-id <id> --raga-id <id> \
    [--tala <tala>] [--language <lang>] \
    [--source-url <url>] [--source-label <label>] \
    [--source-type <type>] [--notes <text>]
```

**Exit codes:** 0 = written or duplicate-skipped. 1 = validation error (file unchanged).

**Output prefixes:** `[NODE+]` `[EDGE+]` `[EDGE-]` `[EDGE~]` `[YOUTUBE+]` `[RAGA+]` `[RAGA~]` `[COMPOSER+]` `[COMP+]` `[SOURCE+]` `SKIP (duplicate)` `ERROR`

> **When to use `apply_diff` instead of `write_cli.py`:**
> Recording files (`recordings/*.json`), adding `notes` fields to compositions,
> or any operation not yet covered by `write_cli.py`.

## Schema reference

### musicians/{id}.json ‚Äî Node fields

| field | type | meaning |
|---|---|---|
| `id` | string | Snake_case unique key. **Never rename once set.** |
| `label` | string | Display name as the musician is commonly known. |
| `sources` | array | List of source objects (see below). At least one required. |
| `born` | int \| null | Birth year only. `null` if unknown. |
| `died` | int \| null | Death year only. `null` if living or unknown. |
| `era` | enum | See Era vocabulary below. |
| `instrument` | enum | See Instrument vocabulary below. |
| `bani` | string | Stylistic school / lineage label. Free text. |
| `youtube` | array | List of extended recording objects. May be empty `[]`. |

### Source object

```json
{
  "url":   "https://en.wikipedia.org/wiki/Tyagaraja",
  "label": "Wikipedia",
  "type":  "wikipedia"
}
```

| field | type | meaning |
|---|---|---|
| `url` | string | Full URL to the source. |
| `label` | string | Human-readable name shown as the link label in the UI. |
| `type` | enum | `wikipedia` ¬∑ `pdf` ¬∑ `article` ¬∑ `archive` ¬∑ `other` |

A node may have multiple sources. The first source is treated as the primary link.
Wikipedia sources should appear first when present.

### YouTube recording object (extended)

```json
{
  "url":            "https://youtu.be/XXXXXXXXXXX",
  "label":          "Composition ¬∑ Raga ¬∑ Tala ‚Äî Artist, Event Year",
  "composition_id": "entharo_mahanubhavulu",
  "raga_id":        "sri",
  "year":           2019
}
```

`composition_id`, `raga_id`, and `year` are optional but should be set whenever
identifiable. They reference IDs in `ragas/{id}.json` and `compositions/{id}.json`.
The `version` field is optional ‚Äî use it when the same musician has more than one
recording of the same composition. Omit when only one version exists.

### musicians/_edges.json ‚Äî Edge fields

| field | type | meaning |
|---|---|---|
| `source` | node id | The **guru** (teacher). |
| `target` | node id | The **shishya** (student). |
| `confidence` | float 0‚Äì1 | How well-sourced is this relationship. |
| `source_url` | string | URL where this relationship is explicitly stated. |
| `note` | string | Optional qualifier on the nature of the relationship. |

### Era vocabulary

| value | meaning |
|---|---|
| `trinity` | The three 18th-century composer-saints. |
| `bridge` | 19th‚Äìearly 20th century figures connecting the Trinity to the modern tradition. |
| `golden_age` | Architects of the modern concert format (~1890‚Äì1950). |
| `disseminator` | Mid-20th century figures who carried the tradition outward. |
| `living_pillars` | Active or recently deceased figures who defined contemporary practice. |
| `contemporary` | Active musicians defining the current era. |

### Instrument vocabulary

`vocal` ¬∑ `veena` ¬∑ `violin` ¬∑ `flute` ¬∑ `mridangam` ¬∑ `bharatanatyam`

New values may be added freely ‚Äî each gets a distinct visual shape in the graph.

### Confidence scale

| range | meaning |
|---|---|
| 0.95‚Äì1.0 | Explicitly stated in Wikipedia infobox or unambiguous prose. |
| 0.85‚Äì0.94 | Clearly implied; cross-confirmed across multiple pages. |
| 0.70‚Äì0.84 | Single prose source, or confirmed 2-hop lineage. |
| below 0.70 | Speculative ‚Äî must carry a `note` explaining the uncertainty. |

### ragas/{id}.json ‚Äî Raga object

| field | type | notes |
|---|---|---|
| `id` | string | snake_case, permanent |
| `name` | string | canonical name |
| `aliases` | array of strings | alternate spellings / names |
| `melakarta` | int \| null | melakarta number (1‚Äì72); for mela ragas = their own number; deprecated on janya ragas (use `parent_raga` instead) |
| `is_melakarta` | bool | `true` if this raga is one of the 72 melakartas; absent or `false` for janya ragas (ADR-021) |
| `cakra` | int \| null | cakra number (1‚Äì12) for melakarta ragas; `null` for janya ragas (ADR-021) |
| `parent_raga` | string \| null | id of parent mela raga if janya; `null` for mela ragas |
| `sources` | array | list of source objects; at least one required |
| `notes` | string | free-text musicological note |

| 3 | Agni | 13‚Äì18 | 9 | Brahma | 49‚Äì54 |
| 4 | Veda | 19‚Äì24 | 10 | Disi | 55‚Äì60 |
| 5 | Bana | 25‚Äì30 | 11 | Rudra | 61‚Äì66 |

### compositions/_composers.json ‚Äî Composer object

| field | type | notes |
|---|---|---|
| `id` | string | snake_case, permanent |
| `name` | string | canonical name |
| `musician_node_id` | string \| null | links to a node in `musicians.json` if the composer is also a lineage node; `null` otherwise |
| `born` / `died` | int \| null | year only |
| `sources` | array | list of source objects; at least one required |

### compositions/{id}.json ‚Äî Composition object

| field | type | notes |
|---|---|---|
| `id` | string | snake_case, permanent |
| `title` | string | canonical title |
| `composer_id` | string | references `composers[].id` |
| `raga_id` | string | references `ragas[].id` |
| `tala` | string | e.g. `adi`, `rupaka`, `misra_chapu` |
| `language` | string | `telugu`, `sanskrit`, `tamil`, `kannada` |
| `sources` | array | list of source objects; may be empty `[]` if no external reference exists |
| `notes` | string | free-text musicological note |

## Recordings ‚Äî `carnatic/data/recordings/`

Each file is a single structured concert recording ‚Äî one bare JSON object per
recording event. See `recordings/READYOU.md` for the full schema and workflow.

To add or edit a recording, use `apply_diff` directly on the relevant
`recordings/{id}.json` file. Then run `python3 carnatic/render.py`.

## Standard workflow for any write session

```
1. python3 carnatic/cli.py stats          # orient
2. Run existence checks before any write  # avoid duplicates
3. python3 carnatic/write_cli.py <cmd>    # atomic write
4. python3 carnatic/cli.py validate       # confirm integrity
5. python3 carnatic/render.py             # rebuild graph.html
```

### YouTube ingestion ‚Äî mandatory pre-checks

Before adding any YouTube link:
1. `python3 carnatic/cli.py url-exists <url>` ‚Äî if FOUND, report and stop.
2. `python3 carnatic/cli.py musician-exists "<artist name>"` ‚Äî note exact id if FOUND; flag if NOT FOUND.
3. `python3 carnatic/cli.py raga-exists "<raga name>"` ‚Äî note exact id if FOUND; add raga first if NOT FOUND.
4. `python3 carnatic/cli.py composition-exists "<composition name>"` ‚Äî note exact id if FOUND; add composition first if NOT FOUND.
5. `python3 carnatic/write_cli.py add-youtube --musician-id <id> --url <url> --label <label> [--raga-id <id>] [--composition-id <id>] [--year <year>]`

## Hard constraints ‚Äî never do these

- **Never edit any `.json` file directly.** Use `write_cli.py` or `apply_diff` via a script.
- **Never rename an existing `id` field** in any file. IDs are permanent keys.
- **Never rename a `musicians/{id}.json` file.** The filename is the node's permanent key.
- **Never create a musician node without at least one entry in `sources`.**
- **Never create a raga without at least one entry in `sources`** (Wikipedia preferred but not required).
- **Never create a composition without a verified `composer_id` and `raga_id`.**
- **Never infer an edge from shared `bani` alone.** Stylistic similarity is not lineage evidence.
- **Never silently drop an unmatched YouTube link.** Always flag it.
- **Never add speculative edges without a `note` field** when `confidence` is below 0.70.
- **Never wrap a musician node file** in an array or outer key. Each `musicians/{id}.json` is a bare object.
- **Never wrap a raga file** in an array or outer key. Each `ragas/{id}.json` is a bare object.
- **Never wrap a composition file** in an array or outer key. Each `compositions/{id}.json` is a bare object.
- **Never wrap a recording file** in `{"recordings": [...]}`. Each file is a bare object.
- **Never set `musician_id`** in a recording to a value that does not exist in `musicians/`.
- **Never edit `graph.json` directly.** It is regenerated by `render.py`.
