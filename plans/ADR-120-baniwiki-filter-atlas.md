# ADR-120: BaniWiki filter atlas — query layer in TiddlyWiki5 filter DSL

**Status**: Proposed
**Date**: 2026-05-08
**Agents**: graph-architect (proposer), carnatic-coder (implementer of generated atlas tiddler)
**Depends on**: ADR-013 (single source of truth traversal), ADR-014 (CLI orientation tools), ADR-017 (phonetic search), ADR-095 (BaniWiki exploration), ADR-117 (Node deployment), ADR-118 (vocabulary), ADR-119 (mapper)
**Companion ADRs**: ADR-121 (phase-out)

---

## Context

A vocabulary (ADR-118) and a mapper (ADR-119) put the data into BaniWiki. They do not yet prove the data is *useful* there. The proof is the query layer: every question we currently ask of `data/**` via Python (`carnatic/cli.py`, `carnatic/graph_api.py`) must be re-expressible as a TiddlyWiki5 filter expression — or its inexpressibility must be named, scoped, and accepted as a known gap.

The user has framed this directly:

> *"All that we need is accomplished via filter operations on tiddlers. Our data itself can be handled between tags and field data."*

That framing is right, and ADR-118's vocabulary is designed against it. This ADR is the **proof of work**: a tiddler-resident, machine-validated catalogue of every existing query and its TW5 filter equivalent. We call it the **Filter Atlas**.

The Atlas serves three audiences:

1. **The Architect**: as a coverage report. If a query has no clean filter equivalent, the vocabulary (ADR-118) needs an addendum.
2. **The Coder**: as a recipe book. When porting a panel from `graph.html` JS to a BaniWiki widget, the Coder reaches for the Atlas before inventing a new query.
3. **The Librarian and the rasika**: as a navigation primer. The Atlas is the BaniWiki entry point for "how do I find X?"

The TiddlyWiki filter DSL is — per `grok_tiddlywiki.html` and the TW5 docs — a small, composable language built around **filter steps** and **operators**. Operators include `tag[X]`, `field:foo[bar]`, `prefix[Y]`, `sort[field]`, `search[term]`, `limit[N]`, `nsort[]`, set ops `+`, `-`, `~`, and many more. Filters compose left-to-right; each step reduces or transforms the running list of titles. Multi-run filters use `+` to chain.

The Atlas demonstrates that our queries are well-served by this language.

### Forces

| Force | Direction |
|---|---|
| **Coverage** | Every public query in `cli.py` and `graph_api.py` MUST be in the Atlas. No silent omissions. |
| **Tested** | Each Atlas entry MUST be runnable inside the served wiki, against the live tiddler tree, and must produce the expected result set. The Atlas is verified, not aspirational. |
| **Idiomatic** | Filters MUST use the documented TW5 operator set first; custom filter operators (added as JS macro tiddlers) are a last resort, named and ADR-justified. |
| **Composable** | Filter expressions are read left-to-right and composed by concatenation. The Atlas favours short, named subfilters (e.g. `\function .musicians.bani(b) [tag[Musician]tag<b>]`) over long inline filters. |
| **Auto-generated where possible** | The Atlas is mostly generated by the mapper from a single source-of-truth list of (Python query → TW5 filter) pairs. Drift is impossible because the Python query and the TW5 filter live in the same record. |
| **Honest about gaps** | The Atlas explicitly catalogues queries that cannot be expressed in pure filter DSL (e.g. ADR-017 phonetic search, the bani-flow trail's multi-key sort). For each gap, the Atlas proposes a remediation path (custom operator, JS macro, or accept as inexpressible). |

## Pattern

**Christopher Alexander, Property 1, *Levels of Scale*; Property 9, *Deep Interlock*.** The Atlas exhibits both: filter steps are the small scale; named subfilters (`.musicians.bani`) are the medium scale; the full Atlas is the large scale. Each level interlocks with the next: a complex filter is a composition of named subfilters, each of which is a composition of operators.

**Pattern 60, *Accessible Green*.** Every neighbourhood needs a green it can walk to. Every BaniWiki user needs a query they can express. The Atlas is the green: a place every user knows is there, with named entries every user can borrow.

**Concept analogue: the Rosetta Stone.** The Atlas is bilingual — Python on one side, TW5 filter DSL on the other. Both columns describe the same query. The Atlas's correctness is its bilinguality: the columns must agree on what they return.

---

## Decision

**Produce a generated, tested, version-controlled Filter Atlas as part of every BaniWiki build.** The Atlas is itself a tiddler (`Atlas/Filters`), generated by the mapper from a single Python source-of-truth.

### 1 — Atlas source: `carnatic/baniwiki/atlas.py`

A single Python module catalogues every query as a record:

```
@atlas_entry(
    section="Musicians",
    title="All musicians of a given bani",
    python_example="python3 carnatic/cli.py musicians --bani semmangudi",
    python_function=cli.musicians_by_bani,
    tw_filter="[tag[Musician]tag[bani:semmangudi]]",
    notes="Replace `semmangudi` with the bani slug.",
)
```

Each entry binds the Python query function to its TW5 filter equivalent. The module is the source; the Atlas tiddler is the projection.

### 2 — Atlas tiddler: `Atlas/Filters`

The mapper (ADR-119) generates `baniwiki/tiddlers/Atlas/Filters.tid`. The tiddler is wikitext (`type: text/vnd.tiddlywiki`), structured as one section per category with a table per section:

```
! Musicians

| Question | Bani Flow CLI | TW5 filter | Try |
|---|---|---|---|
| All musicians of a given bani | `python3 carnatic/cli.py musicians --bani semmangudi` | `[tag[Musician]tag[bani:semmangudi]]` | <$button>Run<$action-navigate $to="…"/></$button> |
| ... | ... | ... | ... |

! Ragas

...
```

The "Try" column embeds a TW5 button widget that runs the filter live and shows the result count. The Atlas is therefore *interactive*: open it in the served wiki, click Try, see the result set update against the current tiddler tree.

### 3 — Atlas verification

A test (`carnatic/baniwiki/tests/test_atlas.py`) iterates every `@atlas_entry`:

- Runs the Python query against `data/**`, captures the result set as a sorted list of IDs.
- Spawns a TW5 process (`tiddlywiki baniwiki/ --output … --rendertiddler …`) that evaluates the TW5 filter, captures the result set as a sorted list of titles (stripped of the type prefix).
- Asserts equality.

This is run on every CI build. **An Atlas entry whose two columns disagree is a build failure.**

### 5 — Named subfilters (TW5 functions, per `grok_tiddlywiki.html` "Functions")

Where a query repeats, the Atlas defines a TW5 **function** (a named filter), stored as a tiddler under `Atlas/Functions/<name>`:

```
\function .musicians.of.bani(b) [tag[Musician]tag<b>]
\function .compositions.of.composer(c) [tag[Composition]tag<c>]
\function .recordings.of.composition(c) [tag[Recording]field:composition_id<c>]
\function .recordings.of.musician(m) [tag[Recording]tag[performer:<m>]]
\function .janyas.of.mela(m) [tag[Raga]tag<m>]
\function .gurus.of(m) [[Musician/<m>]get[guru_ids]enlist-input[]]
```

Functions are imported into the Atlas tiddler via `\import [tag[Atlas/Functions]]` so every Atlas filter and every BaniWiki widget can use them.

### 6 — Gaps (open and named)

The Atlas **does not pretend** every query maps cleanly. Three classes of gap are documented:

**Gap A — Multi-key sorts.** The bani-flow trail (ADR-013) sorts musicians by era, then bani, then chronology. TW5's `sort[]` is single-key, but `nsort[]` and chained sort steps cover this:

```
[tag[Musician]] :sort[era] :sort[bani] :sort[born]
```

This works (TW5 sorts are stable). Documented as expressible.

**Gap B — Phonetic / transliteration search (ADR-017).** TW5 `search[]` is substring-match. The IAST/ITRANS/Roman tri-modal search needs either:

- (preferred) A custom filter operator `phonetic[]` registered as a JS module tiddler, wrapping the existing JS phonetic library.
- (alternative) Pre-computed search-key fields on every tiddler (`label_iast`, `label_itrans`, `label_roman`), so `[search:label_iast[X]]` works.

The Atlas catalogues this gap and proposes both remediations. Decision deferred to implementation.

**Gap C — Graph traversals beyond two hops.** "All musicians within three guru-hops of Tyagaraja" requires recursion. TW5 has the `:then` and `:else` filter run prefixes and the recursive filter pattern, but multi-hop graph traversal is most cleanly done via a custom JS macro that calls `$tw.wiki.filterTiddlers` in a loop. The Atlas catalogues this and recommends a `Macros/GraphTraversal/` subdirectory of macro tiddlers.

### 7 — Atlas as living document

The Atlas is regenerated on every `bani-wiki-build`. New entries are added to `atlas.py`; the tiddler refreshes automatically. PRs that add a new query function but no Atlas entry are caught by a CI check (`every public query function MUST appear in atlas.py`).

## Consequences

### Positive

- **Expressivity is proven, not asserted.** The Atlas runs in CI; every entry is verified against the live tiddler tree. We never wonder whether a query is portable.
- **Recipe book for porting.** Coders porting `graph.html` panels to BaniWiki widgets reach for the Atlas. The Atlas reduces porting from "invent a filter" to "look up a filter."
- **Vocabulary feedback loop.** When a query has no clean filter, the Atlas surfaces the gap. The gap drives an ADR-118 addendum (new tag namespace, new lifted field, new function). The vocabulary improves under pressure.
- **Public navigation aid.** The Atlas tiddler is publicly URL-addressable (per ADR-117). External users learn BaniWiki by reading it.

### Negative / accepted tradeoffs

- **Atlas maintenance burden.** Every new public query needs an entry. Mitigated by the CI check that fails when a query function lacks an entry.
- **Custom filter operators add a JS surface.** The phonetic operator and the graph-traversal macro require maintaining JS code inside BaniWiki. Acceptable: TW5's macro/filter-operator API is small and stable.
- **The Atlas grows large.** Hundreds of entries possible. TW5's table rendering handles this; the tiddler is also indexable by search.

### Risks (and mitigations)

- **Atlas verification drift.** A test that's slow or flaky gets disabled. *Mitigated* by running the Atlas test in parallel and capping runtime per entry; if an entry is too slow, that's a signal the filter is wrong.
- **Two languages of truth (Python and TW5 DSL).** Both expressing the same query. *Mitigated* by the verification step asserting equivalence on the live corpus.
- **Phonetic gap remains unfilled.** ADR-017 phonetic search is a key BaniWiki feature; if neither remediation works, BaniWiki ships with weaker search than `graph.html`. *Mitigated* by treating the phonetic operator as a Phase-2 ADR-119 deliverable, not optional.

## Implementation

### Phase 0 — Acceptance

1. **Architect** (this ADR): authored.
2. **Coder review**: confirm the `@atlas_entry` decorator approach, confirm the verification test is feasible (TW5 CLI render of a filter in CI).
3. **User**: marks `Accepted`.

### Phase 1 — Atlas skeleton (Coder)

4. Implement `carnatic/baniwiki/atlas.py` with the `@atlas_entry` decorator and the inaugural catalogue (§4 of this ADR, expanded to every existing query).
5. Implement Atlas tiddler generator inside the mapper (`forward.py`); the Atlas tiddler is one of the outputs of `forward(data)`.

### Phase 2 — Verification test (Coder)

6. Implement `tests/test_atlas.py` that runs each entry's Python query and TW5 filter and asserts equality.
7. Wire into CI as a hard gate.

### Phase 3 — Functions and gaps (Coder + Architect)

8. Implement `Atlas/Functions/*` named subfilters per §5.
9. Implement the phonetic filter operator (Gap B, preferred remediation) as a JS macro tiddler.
10. Implement the graph-traversal macro (Gap C) as a JS macro tiddler.
11. For any gap that resists remediation, the Architect writes a tagged Open Question on this ADR and on `.clinerules`.

### Phase 4 — Living maintenance

12. CI check: every public function in `cli.py` and `graph_api.py` has a matching `@atlas_entry`.
13. PR template: contributors adding a query function are reminded to add an Atlas entry.

## Open questions

1. **`guru_ids` storage**: as a TW5 field (`guru_ids: id1 id2 id3` space-separated), as multiple `guru:<id>` tags, or both? Both is most flexible (field for ordered access, tag for filter joins) and matches ADR-118's discipline. Confirmed: both.
2. **The bani-flow trail's exact ordering**: er-then-bani-then-chronology (ADR-013) — chained sort steps suffice. Confirmed expressible.
3. **Phonetic search remediation**: custom operator (preferred) vs pre-computed fields (fallback). Decision in Phase 3.
4. **Atlas tiddler size**: hundreds of entries. May need to split (`Atlas/Filters/Musicians`, `Atlas/Filters/Ragas`, …) for browseability. Decision in Phase 1.
5. **Filter testing infrastructure**: TW5 CLI `--rendertiddler` with a filter-printing template is the verification path. Confirm performance is acceptable in CI.
6. **Filter operator portability**: a custom phonetic operator is JS code. It must be vendored into `baniwiki/plugins/phonetic-search/` as a TW5 plugin. Plugin packaging is documented; non-trivial but tractable.

---

## Closing note

A vocabulary without a query layer is a dictionary nobody reads. The Atlas is the index that makes the vocabulary useful. Every query we ask, the Atlas knows; every query the Atlas does not know, we have not yet learned how to ask.

> *"In BaniWiki, a question is a filter; a filter is a tiddler; a tiddler is a question answered."*
